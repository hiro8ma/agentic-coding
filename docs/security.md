# Claude Code セキュリティ設計

## Anthropic のセキュリティ認証

Anthropic は以下のセキュリティ認証を取得しており、エンタープライズ利用の前提条件を満たしている。

| 認証 | 内容 |
|---|---|
| **SOC 2 Type II** | セキュリティ・可用性・機密性の統制が継続的に有効であることを第三者監査で証明 |
| **ISO 27001** | 情報セキュリティマネジメントシステム（ISMS）の国際標準 |
| **ISO 42001** | AI マネジメントシステムの国際標準。AI固有のリスク管理フレームワーク |
| **CSA STAR** | クラウドセキュリティの成熟度評価。クラウドサービスプロバイダ向け |
| **HIPAA** | 医療情報の保護基準。ヘルスケア領域での利用を可能にする |

**面接で語れるポイント**: SOC 2 Type II + ISO 42001 の組み合わせは、従来のセキュリティ統制に加えて AI 固有のリスク（幻覚、バイアス、データ漏洩）も管理していることを意味する。エンタープライズ導入の稟議で「認証は何を取っているか」は必ず聞かれる。

## 4層防御モデル

Claude Code は「多層防御（Defense in Depth）」の考え方で設計されている。1つの層が突破されても次の層で防御する。

### Layer 1: 組織レベル認証

- Anthropic アカウントの認証・認可
- 組織単位のアクセス制御とポリシー設定
- API キーのスコープ管理（どのモデルに、どの機能でアクセスできるか）

### Layer 2: ファイルアクセス制御

- **読み取り専用がデフォルト**: ファイルの読み取りは許可されるが、書き込みは明示的な許可が必要
- **ディレクトリ分離**: プロジェクトルート外のファイルへのアクセスを制限
- **`.gitignore` 準拠**: `.env` やシークレットファイルはデフォルトで除外対象

**設計思想**: 最小権限の原則（Principle of Least Privilege）。Claude Code はデフォルトでは何も書き込めない。ユーザーが明示的に許可した操作だけが実行される。

### Layer 3: コマンド実行制御

- **インジェクション検出**: シェルコマンドに対するインジェクション攻撃を検出
- **ブロックリスト**: 危険なコマンドパターンをブロック（後述）
- **フェイルクローズド（Fail Closed）**: 判断に迷った場合は「拒否」がデフォルト。許可ではなく拒否に倒す

**フェイルクローズド vs フェイルオープン**:

| 方式 | 動作 | 適用場面 |
|---|---|---|
| フェイルクローズド | 不明な入力 → 拒否 | セキュリティ（Claude Code はこちら） |
| フェイルオープン | 不明な入力 → 許可 | 可用性重視のシステム |

### Layer 4: プロンプトインジェクション対策

- **権限システム**: ツール呼び出しごとにユーザー承認を要求（Auto-accept していない場合）
- **コンテキスト分離**: サブエージェントごとに独立したコンテキストウィンドウを持ち、一方が汚染されても他方に波及しない
- **サニタイゼーション**: 外部から取得したテキスト（Web検索結果、ファイル内容）に埋め込まれた悪意ある指示を無効化

**面接で語れるポイント**: プロンプトインジェクションは LLM ベースのツールにとって最大の脅威。Claude Code の対策は「権限システム + コンテキスト分離 + サニタイゼーション」の3つを組み合わせた多層防御になっている。特にサブエージェントのコンテキスト分離は、1つのエージェントが汚染されても被害を局所化する設計。

## データ保持ポリシー

Claude Code が処理したデータの保持期間はプランによって異なる。

| プラン | 条件 | 保持期間 |
|---|---|---|
| 無料 / Pro / Max | モデル改善への使用を許可 | 最大 5 年間 |
| 無料 / Pro / Max | モデル改善への使用を不許可 | 30 日間 |
| Team / Enterprise / API | デフォルト | 30 日間（構成可能） |

### 設定変更方法

`/privacy-settings` コマンドで現在の設定を確認・変更できる。

```
/privacy-settings
```

**重要**: Team / Enterprise / API プランでは、デフォルトでモデルトレーニングにデータが使用されない。個人プランの場合は明示的にオプトアウトする必要がある。

## /security-review コマンド

コミット前にセキュリティ分析を実行するビルトインコマンド。

```
/security-review
```

### チェック項目

| カテゴリ | チェック内容 |
|---|---|
| **SQL Injection** | パラメータ化されていないクエリの検出 |
| **XSS** | ユーザー入力のエスケープ漏れ |
| **認証欠陥** | 認証・認可チェックの欠落 |
| **データ処理** | 個人情報や機密データの不適切な取り扱い |
| **依存性脆弱性** | 既知の脆弱性を含むパッケージの使用 |

**活用方法**: CI/CD パイプラインに組み込むのではなく、開発者がコミット前にローカルで実行するワークフロー。CLAUDE.md に「コミット前に `/security-review` を実行」とルールを書くことで、チーム全体のセキュリティレベルを底上げできる。

## コマンドブロックリスト

Claude Code は特定のコマンドを自動実行しない。ユーザーが許可設定（`allowedTools`）に追加しても、以下のコマンドは自動では実行されず、必ずユーザー承認を求める。

- `curl` — 外部へのデータ送信リスク
- `wget` — 外部からの悪意あるファイルダウンロードリスク

**設計意図**: ネットワークアクセスを伴うコマンドは、データ漏洩（Exfiltration）の経路になりうる。プロンプトインジェクションによって「内部のコードを外部サーバーに送信する」攻撃を防ぐため、ネットワーク系コマンドは常にユーザー承認を必須にしている。

## まとめ: セキュリティ設計の原則

Claude Code のセキュリティ設計は以下の原則に基づいている:

1. **多層防御**: 1つの層が突破されても次の層で防御
2. **最小権限**: デフォルトでは何もできない。明示的に許可された操作だけを実行
3. **フェイルクローズド**: 迷ったら拒否
4. **コンテキスト分離**: エージェント間の汚染を防止
5. **透明性**: データ保持ポリシーを明示し、ユーザーが制御可能
