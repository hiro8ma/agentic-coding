# ループ設計チェックリスト

自動化ループ（定期実行でエージェントに指示を出す仕組み）を作る前に確認する 10 観点。
cobusgreyling/loop-engineering の loop-design-checklist を翻案した。
背景は [[loop-engineering]] を参照。

## 1. 目的とスコープ

- [ ] ゴールが 1 つに絞れている
- [ ] やらないこと（non-goals）を明記した
- [ ] 監視対象の範囲（パス・リポジトリ）を限定した
- [ ] 段階展開の計画がある（L1 レポートのみ → L2 支援修正 → L3 無人化）

## 2. スケジューリング

- [ ] 実行間隔をパターンのコストに合わせて選んだ
- [ ] 初回は即時実行して動作を確認できる
- [ ] スケジュールが永続化されている（セッション終了で消えない）
- [ ] 業務時間外の挙動を決めた
- [ ] 不要になったスケジュールの掃除方法がある

## 3. スキル

- [ ] トリアージ用スキルがある
- [ ] 実行系スキルがプロジェクトの規約に従っている
- [ ] スキルの description が具体的（いつ使うかが読み取れる）
- [ ] ビルド・テストコマンドがドキュメント化されている

## 4. Maker / Checker 分離

- [ ] Implementer と Verifier が別エージェントである
- [ ] **Implementer は自分の作業を完了扱いにできない**
- [ ] Verifier は隔離環境（worktree）でテストを実行する
- [ ] **停止条件の判定は新しいモデル（別コンテキスト）で行う**

## 5. 状態管理

- [ ] 状態ファイル（STATE.md）の形式が文書化されている
- [ ] ループは前回の状態を読んでから動く
- [ ] ループは結果を状態に書き戻す
- [ ] 解決済み項目を剪定する
- [ ] 人間による上書き（override）が記録される

## 6. 人間へのハンドオフ

- [ ] エスカレーション条件が明示されている
- [ ] **denylist パスがある（auth / payments / secrets / infra は自動変更禁止）**
- [ ] 通知ルールが決まっている
- [ ] エスカレーション先（inbox）の場所が決まっている

## 7. コネクタ（MCP）

- [ ] 最小権限で設定されている（読み取り専用と提案用を分ける）
- [ ] PR / チケットの作成・更新ができる
- [ ] bot としてのアイデンティティが明確（人間のアカウントを使わない）

## 8. コストと上限

- [ ] トークン予算を見積もった
- [ ] loop-budget.md に上限を書いた
- [ ] loop-run-log.md に実行を記録する
- [ ] **1 項目あたりの最大イテレーション数を決めた**
- [ ] **1 日あたりの自動 PR 上限を決めた**
- [ ] 一時停止・強制終了の基準を定義した

## 9. 観測

- [ ] 毎回の実行がログに残る
- [ ] 成功指標を選んだ
- [ ] チームの誰でも状態ファイルを確認できる

## 10. 安全

- [ ] allowlist なしの自動マージをしない
- [ ] シークレットが denylist に入っている
- [ ] flaky テストの扱いを文書化した
